029__mechanism_of_reset_and_checkout # In empty new directory, run git init vim f1.txt a git add f1.txt git commit -m "1" # ====================================================================== vim f1.txt a b git commit -am "2" # ====================================================================== vim f1.txt a b c git commit -am "3" # ====================================================================== vim f1.txt a b c d git commit -am "4" # ====================================================================== git log Suppose you want to go back to "commit 3" and you want to cancel (or delete) "commit 4" # ====================================================================== Copy commit ID of "commit 3" git reset --hard commit_3_id git log # ====================================================================== refs/heads/master file contains contents which point to latest commit in master branch in which you're in # ====================================================================== Workflow Git opens ./.git/HEAD file Git opens refs/heads/master file Git knows latest commit # ====================================================================== Reset is the act of change point refernce to the commit # ====================================================================== Where is "commit 4"? Git generally doesn't delete files except for very special cases where garbage collector runs on too many files, which happnes rarely. # ====================================================================== What if you cancel "reset"? Let's see ./.git/ORIG_HEAD file It has "commit 4" ./.git/ORIG_HEAD file writes latest commit in current branch before performing dangerous command # ====================================================================== Let's see ./.git/logs/refs/heads/master file ./.git/logs/refs/heads/master file writes various actions happened in master branch # ====================================================================== # You can cancel "reset" git reset --hard ORIG_HEAD git log # ====================================================================== Better way than using ORIG_HEAD is to see logs/refs/heads/master where more and various data are written # To use logs/refs/heads/master file git reflog # ====================================================================== git checkout commit_id git branch git log # ====================================================================== If you see gistory, changed file is HEAD file Previously, HEAD file pointed to refs/heads/master file current HEAD file points to commit id Your HEAD points to commit than branch This status is "detached" git checkout master
